# AHB-to-APB-Bridge
  This project presents the design and implementation of an AMBA-based AHB to APB Bridge, a critical component in System-on-Chip (SoC) architectures. As modern integrated circuits demand a balance between high performance and low power consumption, the communication between high-speed system components and lowbandwidth peripherals becomes a primary bottleneck.\
  The proposed bridge functions as an interface between the Advanced Highperformance Bus (AHB), which manages high-speed components like processors and on-chip memory, and the Advanced Peripheral Bus (APB), which controls powersensitive peripherals such as UARTs, GPIOs, and Timers.\
  The design is implemented using Verilog HDL, simulated for functional correctness using **Iverilog [ICARUS verilog]**, and verified against the AMBA 3.0/4.0 specifications. Results demonstrate that the bridge successfully maintains data integrity while providing the necessary buffering and handshaking logic required for seamless inter-bus communication.
  
# INTRODUCTION
In modern System-on-Chip (SoC) architectures, the Advanced Microcontroller Bus Architecture (AMBA) protocol is the industry standard for on-chip communication.\
This project focuses on the design of an AHB to APB Bridge, a critical interface component that connects two distinct bus domains: the Advanced High-performance Bus (AHB) and the Advanced Peripheral Bus (APB).\
The AHB is designed for high-performance, high-clock-frequency system modules. In this design, the AHB Slave module interfaces directly with the AHB master
The APB is optimized for minimal power consumption and reduced interface complexity, making it ideal for low-bandwidth peripherals.\
The bridge serves as a protocol converter. It captures the pipelined address and data from the AHB , decodes the target peripheral , and then initiates the slower, nonpipelined transaction on the APB. By using a Finite State Machine (FSM), the bridge ensures that the high-speed AHB master can communicate with slower peripherals without violating timing constraints or causing system-wide stalls.

# AHB SLAVE MODULE

The AHB slave module is designed for high-performance, high-clock-frequency system modules. It supports features like burst transfers and pipelined operations.\
The AHB slave takes the following as inputs :
1.	Hclk: The system clock that synchronizes all bus operations.
2.	Hesetn: An active-low reset signal used to initialize the slave registers to zero.
3.	Hwrite: Indicates the direction of the transfer; logic high for a write and low for a read.
4.	Hreadyin: A signal from the system indicating that the previous bus transfer has completed.
5.	Htrans: Indicates the type of the current transfer, such as Non-Sequential or Sequential.
6.	Haddr: The 32-bit system address bus used to identify the target memory location.
7.	Hwdata: The data bus used to transfer data from the master to the slave during a write.
8.	Prdata: The data read from the APB peripheral, which is passed back to the AHB bus.
\
And the following as outputs:
1.	Valid: A control signal generated by the slave to notify the bridge that a valid transfer is requested.
2.	Hwritereg: A registered version of the hwrite signal used to align the write control with the data phase.
3.	Hresp: The transfer response signal, fixed at 2’b00 to indicate an “OKAY” status.
4.	Tempselx: A decoded select signal used to choose one of three specific APB peripherals.
5.	Haddr1: The first stage of the pipelined address register.
6.	Haddr2: The second stage of the pipelined address register.
7.	Hwdata1: The first stage of the pipelined write data register.
8.	Hwdata2: The second stage of the pipelined write data register.
9.	Hrdata: The output data bus that provides read data back to the AHB master.

**Assigning address**\
	The address is assigned at positive edge of hclk. Since the system is active-low reset, the addresses are set to 0 when hresetn is 0.Else the addresses are assigned using blocking statements while haddr is assigned to haddr1 and haddr1 is assigned to haddr2.

**Assigning write data**\
	The data is assigned at positive edge of hclk. Since the system is active-low reset, the data are set to 0 when hresetn is 0.Else the addresses are assigned using blocking statements while hdata is assigned to hdata1 and hdata1 is assigned to hdata2.

**Peripheral slave selection**\
	This process work on the address range given and it strictly follow the peripheral memory map.\
  - When the range of address is 32’h8000_0000 to 32’h8400_000, 001 is selected.
  - When the range of address is 32’h8400_0000 to 32’h8800_000, 010 is selected.
  - When the range of address is 32’h8800_0000 to 32’h8c00_000, 100 is selected.

**Valid**\
	The valid signal is enabled when the address given is on the range of peripheral memory and also checks whether the transition is Non sequential or sequential i.e., when htrans is 2’b10 or 2’b11. Else the valid signal is disabled.

**Hwritereg**\
	This signal is enabled only when the hwrite and hresetn signal is enabled. Else it is disabled.

**Hrdata**\
	This is the peripheral read data which is sent by the APB controller in response to the low hwrite and the tempselx signal.

**Hresp**\
	The transfer response signal, fixed at 2’b00 to indicate an “OKAY” status.
 
# APB CONTROLLER MODULE

This module contains a Finite State Machine (FSM) that manages the protocol conversion.The FSM moves through eight states, including idle, read, renable, wwait, write, and wenable.\
The APB controller takes output of AHB slave as input and they are:
1.	Hclk: The system clock that synchronizes the controller’s internal state machine transitions.
2.	Hresetn: An active-low reset signal that initializes the state machine to the idle state.
3.	Valid: A control input from the AHB slave indicating a valid transaction is ready for the APB bus.
4.	Hwritereg: An input signal indicating whether the current transaction is a write (high) or a read (low).
5.	Haddr1: The first stage of the pipelined address used for standard write operations.
6.	Haddr2: The second stage of the pipelined address used during back-to-back write operations.
7.	Hwdata1: The first stage of the pipelined write data to be driven onto the APB bus.
8.	Hwdata2: The second stage of the pipelined write data used for concurrent AHB-APB write transfers.
9.	Prdata: The data bus carrying information from the selected APB peripheral to the bridge.
10.	Tempselx: Decoded selection bits that determine which specific pselx line to assert.
\
And the outputs of APB controller are:
1.	Pwrite: The APB direction signal; high for peripheral writes and low for peripheral reads.
2.	Penable: The APB strobe signal used to indicate the Access phase of an APB transfer.
3.	Pselx: The peripheral select lines used to activate a specific slave on the APB bus.
4.	Pwdata: The 32-bit data bus that provides data from the bridge to the APB peripherals.
5.	Paddr: The APB address bus used to communicate the target register location to the peripheral.

**State declaration**\
	The states are declared using parameter for the Finite State Machine. Two registers “present” and “next” are declared for internal calculation of Finite State Machine.
There are 8 states that are present in the Finite State Machine. They are:
1.	IDLE
2.	WWAIT
3.	WRITEP
4.	WENABLEP
5.	WRITE
6.	WENABLE
7.	READ
8.	RENABLE

**Present state logic**\
	The present state is set to idle if hresetn is low since it is active-low reset. Else it is set to next state.

**Next state logic**\
	The next state logic is build using case statement. This is an fsm which functions depending on valid and hwritereg.
- At IDLE state: This is the default reset state. If the valid signal is high and hwritereg is asserted, it moves to wwait for a write operation. If valid is high but hwritereg is low, it transitions directly to read
-	At WWAIT state: Used specifically for AHB write operations to align with the AHB data phase. If valid remains high, it moves to writep to handle a pipelined transfer; otherwise, it moves to the standard write state.
-	At WRITEP state: Represents the Setup Phase for a pipelined write operation. This state ensures that the second buffered address (haddr2) and data (hwdata2) are used for the APB transfer. It transitions unconditionally to wenablep.
-	At WENABLEP state: Represents the Access Phase for pipelined transfers where penable is asserted. Based on the valid and hwritereg status, it can loop back to writep for more pipelined data, return to a standard write, or move to wenable.
-	At WRITE state: Represents the Setup Phase for a standard write operation. If valid is detected, it moves to wenablep to process the current write while preparing for the next; if not, it moves to wenable.
-	At WENABLE state: Represents the Access Phase for a standard write operation. After completion, it checks valid to determine if it should move to wwait for a new write, read for a new read, or back to idle.
-	At READ state: Represents the Setup Phase for a read operation. The FSM unconditionally moves to the renable state in the next clock cycle to complete the access.
-	At RENABLE state: Represents the Access Phase for a read operation. Upon completion, the FSM determines the next state based on whether a new read or write request is pending (valid), or returns to idle.
-	Default is given to idle state.
**State functionality**\
	In this block, we define what each state does.
-	READ: The bridge initiates the Setup Phase for a read. It asserts pselx using the decoded tempselx and drives the address from haddr1 onto paddr, while keeping penable low and pwrite at 0.
-	RENABLE: This is the Access Phase for a read. The bridge maintains the pselx and paddr from the setup phase but asserts penable (set to 1) to signal the peripheral to provide data.
-	WWAIT: This is a synchronization state where the bridge waits for the AHB data phase to catch up. During this time, the APB interface remains inactive with pselx and penable set to 0.
-	WRITE: The Setup Phase for a standard write operation. It drives the first pipelined address (haddr1) to paddr, sets pwrite to 1, and places the first pipelined data (hwdata1) onto the pwdata bus.
-	WRITEP: The Setup Phase for a pipelined write. Similar to the standard write, but it uses the second stage of pipelined registers (haddr2 and hwdata2) to drive the APB bus, allowing back-to-back AHB transfers to be processed.
-	WENABLE: The Access Phase for a standard write. The bridge maintains the paddr, pwdata, and pwrite values from the write state while asserting penable to 1 to complete the transfer to the peripheral.
-	WENABLEP: The Access Phase for a pipelined write. It asserts penable while holding the secondary pipelined data (hwdata2) and address (haddr2) steady for the duration of the cycle.

# BRIDGE TOP MODULE
The bridge_top module serves as the structural backbone of the design, acting as the top-level wrapper that integrates the AHB Slave and the APB Controller. It defines the external interface of the bridge and facilitates the internal communication between the two bus protocols.\
The APB controller and AHB slave are instantiated in this module and are connected using wires and output and inputs of each other.
